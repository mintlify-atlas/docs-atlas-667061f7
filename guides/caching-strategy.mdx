title: Caching Strategy
description: Understanding shvm-db's edge caching architecture using Cloudflare Cache API

shvm-db implements a sophisticated edge caching layer using Cloudflare's Cache API to minimize Durable Object invocations and deliver sub-millisecond read latency for frequently accessed data.

## Architecture Overview

The caching strategy employs a **write-through cache** at the edge:

1. **Reads**: Check edge cache first, falling back to Durable Object + SQLite on cache miss
2. **Writes**: Write to Durable Object, then asynchronously populate edge cache
3. **Deletes**: Write tombstone to both Durable Object and edge cache

This approach provides:
- Strong consistency (reads after writes always see latest data)
- Ultra-low latency for cached reads (< 1ms at edge)
- Reduced Durable Object load and cost

## CacheManager Implementation

The `CacheManager` class provides a clean abstraction over Cloudflare's Cache API:

```typescript
// From src/cache.ts:3-14
export class CacheManager {
  private readonly cache: Cache;
  private readonly log: (component: string, ...args: any[]) => void;

  constructor(env?: any) {
    this.cache = (caches as any).default;
    this.log = createLogger(env);
  }

  private getCacheKey(type: "item" | "table", key: string): Request {
    return new Request(`https://shvm-db.local/cache/${type}/${encodeURIComponent(key)}`);
  }
}
```

### Cache Key Structure

Cache keys are namespaced by type (`item` or `table`) and use pseudo-URLs:

```typescript
// Items: https://shvm-db.local/cache/item/user%23123%23default
// Tables: https://shvm-db.local/cache/table/users
```

This structure ensures:
- No collisions between items and table metadata
- URL-safe encoding of keys containing special characters
- Cloudflare-compatible cache key format

## Reading from Cache

### getItem - Type-Safe Reads

```typescript
// From src/cache.ts:16-29
async getItem<T>(type: "item" | "table", key: string): Promise<T | null> {
  const req = this.getCacheKey(type, key);
  try {
    const res = await this.cache.match(req);
    if (res) {
      const data = await res.json() as any;
      if (data && data._deleted) return null;
      return data as T;
    }
  } catch (err) {
    this.log("cache", `Parse error on match for ${type}/${key}:`, err);
  }
  return null;
}
```

### getItemRaw - Detailed Cache Information

For observability, `getItemRaw` returns cache hit status and tombstone detection:

```typescript
// From src/cache.ts:31-45
async getItemRaw(type: "item" | "table", key: string): Promise<{ 
  hit: boolean; 
  deleted: boolean; 
  raw?: string; 
  data?: any 
}> {
  const req = this.getCacheKey(type, key);
  try {
    const res = await this.cache.match(req);
    if (res) {
      const text = await res.text();
      const data = JSON.parse(text);
      if (data && data._deleted) return { hit: true, deleted: true, raw: text, data };
      return { hit: true, deleted: false, raw: text, data };
    }
  } catch (err) {
    this.log("cache", `Parse error on raw match for ${type}/${key}:`, err);
  }
  return { hit: false, deleted: false };
}
```

## Writing to Cache

### putItem - Write-Through Caching

```typescript
// From src/cache.ts:48-58
async putItem(
  ctx: ExecutionContext, 
  type: "item" | "table", 
  key: string, 
  data: any, 
  ttlSeconds: number
): Promise<void> {
  const req = this.getCacheKey(type, key);
  const res = new Response(JSON.stringify(data), {
    headers: {
      "Cache-Control": `max-age=${ttlSeconds}`
    }
  });

  // WaitUntil is required because Cloudflare caches do not naturally await 
  // response streams inside worker isolates that close immediately
  ctx.waitUntil(this.cache.put(req, res));
}
```

<Info>
Using `ctx.waitUntil` allows the response to return to the client immediately while the cache write completes asynchronously.
</Info>

### TTL Configuration

Different data types use different TTLs:

| Type | TTL | Rationale |
|------|-----|----------|
| Items | 300s (5 min) | Balance between freshness and cache efficiency |
| Tables | 7200s (2 hours) | Table schemas rarely change |

## Cache Invalidation

### Tombstone Strategy

Instead of calling `cache.delete()`, shvm-db writes **tombstones** to prevent thundering herd problems:

```typescript
// From src/cache.ts:60-69
async deleteItem(
  ctx: ExecutionContext, 
  type: "item" | "table", 
  key: string, 
  ttlSeconds: number
): Promise<void> {
  // Write a tombstone rather than doing caches.delete() to avoid 
  // thundering-herd on misses
  const req = this.getCacheKey(type, key);
  const res = new Response(JSON.stringify({ _deleted: true }), {
    headers: {
      "Cache-Control": `max-age=${ttlSeconds}`
    }
  });
  ctx.waitUntil(this.cache.put(req, res));
}
```

### Why Tombstones?

Deleting from cache creates a **thundering herd** problem:

<Steps>
  <Step title="Item is deleted from cache">
    `cache.delete()` removes the cached entry.
  </Step>
  
  <Step title="Concurrent requests arrive">
    Multiple requests for the same key arrive at different edge nodes.
  </Step>
  
  <Step title="Cache misses trigger DO invocations">
    Each request invokes the Durable Object, causing a spike in load and latency.
  </Step>
  
  <Step title="Cache is repopulated multiple times">
    Multiple redundant cache writes occur.
  </Step>
</Steps>

**Tombstones prevent this** by caching the absence of data:

<Steps>
  <Step title="Tombstone is written to cache">
    `{ _deleted: true }` is stored with a TTL.
  </Step>
  
  <Step title="Concurrent requests arrive">
    Multiple requests hit the cache.
  </Step>
  
  <Step title="Cache returns tombstone">
    All requests see the tombstone and return immediately without invoking the DO.
  </Step>
  
  <Step title="Tombstone expires naturally">
    After TTL, the cache entry expires and the cycle repeats if needed.
  </Step>
</Steps>

## Integration with Item Operations

### GetItem with Caching

The read path checks cache before invoking the Durable Object:

```typescript
// From src/index.ts:306-337
if (op === "GetItem") {
  const skDef = metadata.KeySchema.find(k => k.KeyType === "RANGE");
  const skVal = skDef ? body.Key[skDef.AttributeName] : undefined;
  const skRaw = getRaw(skVal) || "default";
  const doKey = `${pkRaw}#${skRaw}`;

  const doStart = Date.now() - requestStartTs;
  const { hit, deleted, data } = await cacheManager.getItemRaw("item", doKey);

  let item: any = null;

  if (hit) {
    addEvent("cache_read", doStart, Date.now() - requestStartTs - doStart, 
      { hit: true, type: "item" });
    if (!deleted) item = data;
  } else {
    addEvent("cache_read", doStart, Date.now() - requestStartTs - doStart, 
      { hit: false, type: "item" });
    const doInvokeStart = Date.now() - requestStartTs;
    item = await pStub.handleGetItem(doKey, requestId, Date.now());
    const doEnd = Date.now() - requestStartTs;
    addEvent("do_get_item", doInvokeStart, doEnd - doInvokeStart);

    // Store into cache on miss
    if (item) {
      await cacheManager.putItem(ctx, "item", doKey, item, 300);
    } else {
      await cacheManager.deleteItem(ctx, "item", doKey, 300);
    }
  }

  const resObj = item !== undefined && item !== null ? { Item: item } : {};
  return new Response(JSON.stringify(resObj), { headers: baseHeaders });
}
```

### UpdateItem Cache Invalidation

Updates use a hybrid approach:

```typescript
// From src/index.ts:286-295
else if (op === "UpdateItem") {
  const res: any = await pStub.handleUpdateItem(
    doKey, 
    updates, 
    partitionId, 
    tableName, 
    requestId, 
    body.UpdateExpression, 
    body.ConditionExpression, 
    body.ExpressionAttributeNames, 
    body.ExpressionAttributeValues, 
    body.ReturnValues, 
    Date.now()
  );

  // If returnValues wasn't ALL_NEW, delete from cache to force re-fetch
  if (body.ReturnValues === "ALL_NEW" && res?.Attributes) {
    await cacheManager.putItem(ctx, "item", doKey, res.Attributes, 300);
  } else {
    await cacheManager.deleteItem(ctx, "item", doKey, 300);
  }
  
  return new Response(JSON.stringify(res || {}), { headers: baseHeaders });
}
```

<Info>
When `ReturnValues` is `ALL_NEW`, the cache is updated with the new item. Otherwise, a tombstone is written to force subsequent reads to fetch fresh data from the Durable Object.
</Info>

## DescribeTable Caching

Table metadata is cached separately with a longer TTL:

```typescript
// From src/index.ts:183-196
if (op === "DescribeTable") {
  const cachedSchema = await cacheManager.getItem<TableMetadata>("table", body.TableName);
  if (cachedSchema) {
    addEvent("cache_read", metaStart, Date.now() - requestStartTs - metaStart, 
      { hit: true, type: "table" });
    return new Response(JSON.stringify({ Table: cachedSchema }), { headers: baseHeaders });
  }

  const { metadata } = await metadataService.getTableMetadata(body.TableName);

  // Cache for 2 hours (7200 seconds)
  await cacheManager.putItem(ctx, "table", body.TableName, metadata, 7200);

  return new Response(JSON.stringify({ Table: metadata }), { headers: baseHeaders });
}
```

Before any item operation, the worker fetches table metadata. This metadata is cached to avoid repeated registry lookups:

```typescript
// From src/index.ts:210-224
const cachedSchema = await cacheManager.getItem<TableMetadata>("table", tableName);
if (cachedSchema) {
  metadata = cachedSchema;
  addEvent("get_table_metadata", metaStart, Date.now() - requestStartTs - metaStart, 
    { metadata_source: "cache" });
} else {
  const metaResult = await metadataService.getTableMetadata(tableName);
  metadata = metaResult.metadata;
  await cacheManager.putItem(ctx, "table", tableName, metadata, 7200);
  addEvent("get_table_metadata", metaStart, Date.now() - requestStartTs - metaStart, 
    { metadata_source: "registry" });
}
```

## Performance Impact

### Latency Comparison

| Scenario | Without Cache | With Cache (Hit) | Improvement |
|----------|---------------|------------------|-------------|
| GetItem (hot key) | 3-10ms | < 1ms | 3-10x faster |
| GetItem (cold key) | 3-10ms | 3-10ms (miss) | No change |
| DescribeTable | 2-5ms | < 1ms | 2-5x faster |

### Cost Reduction

Caching reduces Durable Object invocations:

- **90% cache hit rate**: 10x reduction in DO compute costs
- **50% cache hit rate**: 2x reduction in DO compute costs
- **0% cache hit rate**: No cost reduction (but latency still benefits from edge proximity)

### Cache Hit Rate Factors

<Steps>
  <Step title="Access patterns">
    Repeated reads of the same items increase hit rates.
  </Step>
  
  <Step title="Key cardinality">
    Fewer unique keys mean higher hit rates.
  </Step>
  
  <Step title="Write frequency">
    Frequent writes invalidate cache, reducing hit rates.
  </Step>
  
  <Step title="Geographic distribution">
    Cloudflare caches per-POP, so geographically distributed traffic may see lower hit rates.
  </Step>
</Steps>

## Best Practices

<Steps>
  <Step title="Design for cache-friendly access patterns">
    Structure your data model to favor repeated reads of the same keys.
  </Step>
  
  <Step title="Monitor cache hit rates">
    Use the trace API to track `cache_read` events and optimize accordingly.
  </Step>
  
  <Step title="Consider TTL tradeoffs">
    Longer TTLs improve cache efficiency but may serve stale data longer.
  </Step>
  
  <Step title="Use conditional writes to avoid stale updates">
    Even with caching, conditional expressions ensure consistency.
  </Step>
  
  <Step title="Test cache behavior in staging">
    Cloudflare's cache behavior may differ between development and production.
  </Step>
</Steps>

## Debugging Cache Issues

### Trace Headers

Every response includes a `X-SHIVAM-DB-Trace-Summary` header:

```
X-SHIVAM-DB-Trace-Summary: total_ms=1.2,cache_read_ms=0.3,cache_read_hit=true
```

Use this to diagnose:
- Cache hit/miss rates
- Cache read latency
- Total request latency breakdown

### Trace API

For detailed traces, use the `/api/trace` endpoint:

```bash
curl "https://your-worker.workers.dev/api/trace?requestId=abc-123"
```

Response includes all trace events:

```json
{
  "requestId": "abc-123",
  "events": [
    { "step": "cache_read", "startMs": 0, "durationMs": 0.3, "attributes": { "hit": true, "type": "item" } },
    { "step": "request", "startMs": 0, "durationMs": 1.2, "attributes": { "cold_path": true } }
  ]
}
```
