shvm-db uses a rigorous compatibility testing framework that runs every operation against both shvm-db and DynamoDB Local (the "oracle"), comparing responses to ensure exact API compatibility.

## Testing Philosophy

The test suite validates that shvm-db behaves **identically** to DynamoDB:

- Same response structure
- Same error messages and types
- Same edge case behavior
- Same validation rules

This approach ensures developers can use shvm-db as a drop-in replacement without behavioral surprises.

## Test Harness Architecture

The `executeAgainstBoth` function is the core of the testing framework:

```typescript
// From compat-test/src/testHarness.ts:35-61
export async function executeAgainstBoth(
  commandFactory: (client: DynamoDBClient) => Promise<any>
) {
  const { oracleClient, testClient } = getClients();

  let oracleResult, testResult;
  let oracleError, testError;

  try {
    oracleResult = await commandFactory(oracleClient);
  } catch (e) {
    oracleError = e;
  }

  try {
    testResult = await commandFactory(testClient);
  } catch (e) {
    testError = e;
  }

  if (oracleError || testError) {
    // If one errors and the other doesn't, we want compareError to fail
    compareError(oracleError, testError);
  } else {
    compare(oracleResult, testResult);
  }
}
```

### Client Configuration

Two clients are configured to point at different endpoints:

```typescript
// From compat-test/src/testHarness.ts:15-33
export function getClients() {
  if (!oracleClient || !testClient) {
    let oraclePort = parseInt(process.env.ORACLE_PORT || "8000", 10);
    let testPort = parseInt(process.env.TEST_PORT || "8787", 10);

    try {
      const portsStr = fs.readFileSync(
        path.resolve(__dirname, "../.test-ports.json"), 
        "utf8"
      );
      const parsed = JSON.parse(portsStr);
      if (parsed.oraclePort) oraclePort = parsed.oraclePort;
      if (parsed.testPort) testPort = parsed.testPort;
    } catch (e) {
      // ignore
    }

    oracleClient = createTestClient(oraclePort, "/");
    testClient = createTestClient(testPort, "/api/");
  }
  return { oracleClient, testClient };
}
```

### Pagination Tests

```typescript
// From compat-test/src/tests/table.spec.ts:53-95
test("ListTables - Pagination with Limit", async () => {
  await executeAgainstBoth(async (client) => {
    const tableA = `test-list-table-${randomUUID()}`;
    const tableB = `test-list-table-${randomUUID()}`;

    await client.send(new CreateTableCommand({
      TableName: tableA,
      KeySchema: [{ AttributeName: "pk", KeyType: "HASH" }],
      AttributeDefinitions: [{ AttributeName: "pk", AttributeType: "S" }],
      BillingMode: "PAY_PER_REQUEST",
    }));
    await client.send(new CreateTableCommand({
      TableName: tableB,
      KeySchema: [{ AttributeName: "pk", KeyType: "HASH" }],
      AttributeDefinitions: [{ AttributeName: "pk", AttributeType: "S" }],
      BillingMode: "PAY_PER_REQUEST",
    }));

    const res1 = await client.send(new ListTablesCommand({ Limit: 1 }));

    let nextTablesResult;
    if (res1.LastEvaluatedTableName) {
      nextTablesResult = await client.send(new ListTablesCommand({
        Limit: 1,
        ExclusiveStartTableName: res1.LastEvaluatedTableName
      }));
    }

    // Clean up
    await client.send(new DeleteTableCommand({ TableName: tableA }));
    await client.send(new DeleteTableCommand({ TableName: tableB }));

    return {
      res1NamesLength: res1.TableNames?.length,
      res1HasLastEval: !!res1.LastEvaluatedTableName,
      res2NamesLength: nextTablesResult?.TableNames?.length,
    };
  });
});
```

## Error Comparison

Error messages and types must match exactly:

```typescript
// From compat-test/src/diff/comparator.ts
export function compareError(oracleError: any, testError: any) {
  if (!oracleError && !testError) return;
  
  if (oracleError && !testError) {
    throw new Error(`Oracle errored but test succeeded: ${oracleError.message}`);
  }
  
  if (!oracleError && testError) {
    throw new Error(`Test errored but oracle succeeded: ${testError.message}`);
  }
  
  // Compare error types and messages
  expect(testError.name).toBe(oracleError.name);
  expect(testError.message).toBe(oracleError.message);
}
```

## Continuous Integration

The test suite runs on every commit to ensure:

1. **No regressions**: Existing functionality remains compatible
2. **New features match DynamoDB**: New operations behave identically to AWS
3. **Error handling is consistent**: Edge cases produce the same errors

## Writing New Tests

To add a new test:

<Steps>
  <Step title="Create a test file">
    Add a new `.spec.ts` file in `compat-test/src/tests/`.
  </Step>
  
  <Step title="Use executeAgainstBoth">
    Wrap your commands in `executeAgainstBoth` to run against both implementations.
  </Step>
  
  <Step title="Use real AWS SDK commands">
    Import from `@aws-sdk/client-dynamodb` for full type safety.
  </Step>
  
  <Step title="Test both success and failure paths">
    Include tests for valid operations AND expected errors.
  </Step>
  
  <Step title="Clean up resources">
    Delete test tables in `afterAll` hooks to avoid pollution.
  </Step>
</Steps>

### Example Test

```typescript
import { describe, it, beforeAll, afterAll } from "vitest";
import { CreateTableCommand, DeleteTableCommand, PutItemCommand } from "@aws-sdk/client-dynamodb";
import { executeAgainstBoth } from "../testHarness";

describe("My New Feature", () => {
  const tableName = `test-feature-${crypto.randomUUID()}`;

  beforeAll(async () => {
    await executeAgainstBoth(client => client.send(new CreateTableCommand({
      TableName: tableName,
      KeySchema: [{ AttributeName: "pk", KeyType: "HASH" }],
      AttributeDefinitions: [{ AttributeName: "pk", AttributeType: "S" }],
      BillingMode: "PAY_PER_REQUEST"
    })));
  });

  afterAll(async () => {
    await executeAgainstBoth(client => client.send(new DeleteTableCommand({
      TableName: tableName
    })));
  });

  it("should do something", async () => {
    await executeAgainstBoth(client => client.send(new PutItemCommand({
      TableName: tableName,
      Item: { pk: { S: "test" } }
    })));
  });
});
```

## Best Practices

<Steps>
  <Step title="Test against real DynamoDB Local">
    Never mock the oracle—test against the actual implementation.
  </Step>
  
  <Step title="Use random table names">
    Avoid test pollution by generating unique table names.
  </Step>
  
  <Step title="Test error paths explicitly">
    Don't just test happy paths—validate error behavior.
  </Step>
  
  <Step title="Keep tests isolated">
    Each test should create and clean up its own resources.
  </Step>
  
  <Step title="Document expected behavior">
    Use comments to explain why certain errors are expected.
  </Step>
</Steps>
