# Welcome to shvm-db

shvm-db is an experimental DynamoDB-compatible database that rebuilds DynamoDB's core model from first principles using modern serverless primitives. It's designed as a learning tool to demystify how partitioned databases work at scale.

<Note>
  This is an educational project, not production-ready software. Use it to learn how distributed databases work, not to store mission-critical data.
</Note>

## What makes shvm-db unique

DynamoDB proves that a **simple API + brutal operational discipline** can scale to absurd throughput. shvm-db demystifies this by rebuilding the core model using:

- **Cloudflare Durable Objects** for single-writer partitions
- **SQLite** as the per-partition storage engine
- **A unified global Table Registry** for multi-tenant table management

The result is a system where you can see, touch, and understand every bottleneck that makes DynamoDB tick.

## Core features

### DynamoDB API compatibility

shvm-db implements the core DynamoDB operations you use every day:

- `CreateTable`, `DeleteTable`, `ListTables`, `DescribeTable`
- `PutItem`, `GetItem`, `UpdateItem`, `DeleteItem`
- Full condition expression support
- Partition key and optional sort key support

### Strong consistency guarantees

Every partition key maps deterministically to exactly one Durable Object, giving you:

- **Single-writer guarantee** per partition key
- **Read-after-write consistency** within the same partition
- **Fully serialized writes** with no coordination overhead

### Observable performance

Unlike DynamoDB's black box, you can measure exactly where the bottlenecks are:

- Per-partition throughput ceilings (500-2k writes/sec, 2k-10k reads/sec)
- SQLite lock contention under write-heavy workloads
- Durable Object single-threaded execution limits
- Edge cache hit rates and DO wake patterns

<Info>
  System throughput scales linearly with the number of partitions. 1000 partitions can theoretically handle millions of operations per second.
</Info>

## What you'll learn

By building with or studying shvm-db, you'll gain intuition about:

- How partition-based routing eliminates coordination overhead
- Why hot partitions create hard performance ceilings
- How single-writer consistency models simplify distributed systems
- What tradeoffs DynamoDB makes under the hood

<Warning>
  shvm-db intentionally exposes the same bottlenecks as DynamoDB: hot partitions hit hard ceilings, cross-partition operations are expensive, and there's no magic rebalancing.
</Warning>

## Not a toy, a forge

This isn't about beating DynamoDB in production. It's about:

- **Proving the architecture is reproducible**
- **Understanding where the real bottlenecks are**
- **Building an extensible substrate for experimentation**

Once you finish exploring shvm-db:

- DynamoDB stops feeling magical
- Distributed databases stop being abstract
- You gain intuition that books don't give
